## 3. Missing for v0.2

### BRR sample conversion
No rules or docs cover:
- BRR encoding algorithm (4-bit ADPCM with filter coefficients)
- How to convert WAV → BRR (external tools: BRRtools, snesbrr, or custom)
- The B +21 cent tuning requirement and why it matters (16-sample block alignment)
- Loop point alignment constraints in BRR (must be on 16-sample boundaries)
- Sample rate selection tradeoffs (8000/16000/32000 Hz vs ARAM budget)
- How SNESGSS handles sample import (.gsi instrument files)
- Whether we should build BRR conversion into the Python CLI or use external tools

**Needed:** A rule file or design doc covering BRR conversion pipeline options, sample preparation requirements, and the relationship between WAV source files and SNESGSS instrument definitions.

### RS5K integration
No coverage anywhere of:
- What RS5K is (REAPER's built-in sampler plugin)
- How it could provide per-channel sample preview in REAPER
- Whether RS5K can load BRR-compatible samples for authoring preview
- The gap between "sounds in REAPER" and "sounds on SNES" and how to manage expectations
- Whether sample preview is even a v0.2 goal or deferred further

**Needed:** A decision record on whether RS5K-based preview is in scope, and if so, a rule file for how it integrates with the authoring workflow.

### SNESGSS .gsm format analysis
No coverage of:
- .gsm binary structure (partially documented in SNESGSS readme.txt "Music data format" section, but only the compiled channel data format, not the .gsm project wrapper)
- Whether .gsm can be reverse-engineered from the test_projects/ files
- What would be needed to generate .gsm directly (skipping the GUI)
- The risk/reward tradeoff of .gsm generation vs. staying with MIDI import

**Needed:** A design doc or decision record on whether .gsm reverse engineering is worth attempting in v0.2, or whether the MIDI import path is sufficient for now.

### Drum channel support
Active.md mentions it but no rule file covers:
- How the Lua exporter should handle drum tracks differently from melodic tracks
- The SNESGSS drum channel mapping (ch7=hats, ch8=kick/snare/toms, instruments 10–13)
- How drum notation should appear in REAPER (one track? two tracks for ch7/8? note-to-drum mapping?)
- Whether the JSFX monitor needs drum-aware display
- How the Python validator should check drum channel constraints
- How this interacts with the 8-channel limit (6 melodic + 2 drum = 8)

**Needed:** A decision on drum workflow design, then updates to reaper-lua.md, jsfx.md, and snes-audio.md.

### Tempo mapping
Active.md flags tempo import as unverified. No docs cover:
- Whether SNESGSS respects MIDI tempo meta-events or ignores them
- The relationship between SNESGSS native speed (1–99) and BPM
- Whether we should strip tempo from MIDI and document the manual speed setting, or try to map BPM → speed
- Test procedure for verifying tempo import

**Needed:** Empirical testing + decision record.

### MIDI content validation
The Python validator checks MIDI file existence but doesn't parse it. For v0.2:
- No rule or design doc for MIDI binary parsing in Python
- No specification of what "valid constrained MIDI" means beyond format/track count
- No regression test comparing exported .mid against expected binary content

**Needed:** Update to python-build.md with MIDI parsing scope, and a fixture .mid file for regression testing.

### Hooks
SPEC §10.6 described hooks (PostToolUse lint/test, Stop decision proposal). None were set up. No rule file covers:
- Which hooks to configure
- What commands they should run
- How to avoid hooks that slow down iteration

**Needed:** Decide whether hooks add value at current project size, or defer until the codebase is larger.
